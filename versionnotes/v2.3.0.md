# v2.3.0 - BREAKING CHANGE: ADC Workspace Scopes Enforcement

## ADC Authentication with Workspace Scopes

Previously, `gas-fakes` attempted to bypass Google's block on regular Workspace scopes (Drive, Sheets, Docs, etc.) when using Application Default Credentials (ADC) by substituting them with the `cloud-platform` scope.

However, Google has recently tightened these restrictions. Using the default `gcloud` client ID with regular Workspace scopes (even when relying on `cloud-platform`) is no longer sufficient and is frequently blocked.

### What has changed?
Starting with **v2.3.0**, `gas-fakes` will no longer automatically substitute Workspace scopes for `cloud-platform` when using ADC. Instead, it will **enforce the use of a custom OAuth2 client credential file** if your project requires any of the regular Workspace scopes and you are using `authType: adc`. Previously this only applied if you were using restricted or sensitive scopes.


### Why this change?
To ensure reliable authentication and avoid `403 Access Blocked` errors, you must explicitly authorize your local environment using your own Google Cloud Project's client ID rather than the globally shared `gcloud` CLI client ID.

### Note on Domain-Wide Delegation (DWD)
This change **does not affect users using Domain-Wide Delegation (`authType: dwd`)**. When using DWD, the initial ADC login only requires basic identity scopes (like `cloud-platform`), which are not blocked by Google for the default client. The Workspace scopes are handled internally via the service account's token minting process.

### How to Fix (for `authType: adc`)
If you receive an error about Workspace scopes and ADC, you must:
1.  Create an OAuth2 Client ID (type: "Desktop app") in your Google Cloud Console.
2.  Download the credentials JSON file.
3.  Re-initialize `gas-fakes` and provide the path to this file, or manually run:
    ```bash
    gcloud auth application-default login --client-id-file=path/to/your/credentials.json --scopes=...
    ```

For detailed instructions, see the updated [Workspace Scopes Guide](workspace_scopes.md).

## Feature: Automatic Manifest Emulation

If you are building a "pure" `gas-fakes` project that will only ever run locally on Node.js, you no longer strictly require an `appsscript.json` manifest file to handle authorization. 

If the manifest is missing (or has an empty `oauthScopes` array), `gas-fakes` will now automatically emulate one in memory using the scopes defined in your `.env` file (`DEFAULT_SCOPES` + `EXTRA_SCOPES`). It will also set a default `timeZone` to `America/New_York` (or the value of `GF_TIMEZONE`). Note that for full fidelity with live Apps Script deployments, an `appsscript.json` is still recommended.
